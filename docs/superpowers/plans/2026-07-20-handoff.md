## 3. 环境备忘

### 3.1 go.work 陷阱

`legion/go.work` 的 use 列表**不含** `.claude/worktrees/*`。在 worktree 里所有 go 命令必须加 `GOWORK=off`，主工作区不用。

### 3.2 race detector

两套都可用，**不是二选一**：

- **Windows**：LLVM-MinGW 已在系统 PATH，新终端直接 `go test -race ./...`。**无法重开终端时**（例如 agent 会话里的长驻 shell）显式前置即可，gcc 一直都在：
  `PATH="/c/Users/Administrator/AppData/Local/Microsoft/WinGet/Packages/MartinStorsjo.LLVM-MinGW.MSVCRT_Microsoft.Winget.Source_8wekyb3d8bbwe/llvm-mingw-20260616-msvcrt-x86_64/bin:$PATH" CGO_ENABLED=1 go test -race ./...`
- **WSL**：`export PATH=/opt/go/bin:$PATH GOWORK=off CGO_ENABLED=1 GOOS=linux GOARCH=amd64`

**`GOOS=linux` 不能省** —— WSL 的 go env 里 `GOOS='windows'`，不覆盖会报 `-mthreads` 错误（MinGW 选项，Linux gcc 不认）。

**涉及文件权限或符号链接的改动应以 WSL 结果为准**：`TestReadEventsFailsLoudOnUnreadableDirectory` 在 Windows 上是 `t.Skip`，symlink 测试在 Windows 需开发者模式。

### 3.3 `internal/runtime` 里三个会浪费时间的构造难点

写这个包的错误路径测试前先看这两条，都是 PR-H 现场撞出来的：

- **`Coordinator` 持有的是具体类型 `*task.Scheduler`，不是接口**（`coordinator.go` 的结构体定义），所以没法用桩让 `Transition` 失败。它的三种失败是 `ctx.Err()` / `ErrTaskNotFound` / `ErrInvalidTransition`；被测任务通常存在且合法处于 Running，于是**只剩 ctx** —— 让 `RunTask` 在返回错误前取消 ctx，是唯一确定性的杠杆。
- **`newSubRuntime` 用结构体字面量造子 runtime，绕过 `NewRuntime` 的所有兜底**。给 `Runtime` 加任何带默认值的字段时必须同时改它，否则子 runtime 拿到零值。logger 就是这样：漏继承的话子 runtime 第一次走失败路径直接 panic。
- 测 `RunTask` 的失败学习信号时，事件总线**只能拒 learning 事件**。全拒的话会在更早的 `task_started` 发布处就中止，根本走不到被测路径。

### 3.4 变异测试是本仓库的惯例

新增测试后，把生产代码改回旧写法确认测试**确实失败**，再改回来。

本轮它抓到过两个假通过的测试（详见 `testing.md`）。这一步不能省。

### 两个曾经的误判（避免重蹈）

1. **日志里的 `agent_id` 字段**：成功事件里是实际执行的 agent（可信），**失败事件里是 coordinator 自身的 ID**（恒为 `default-agent`）。我曾据此误判「sub-agent 没生效」，实际生效了。已写进 `troubleshooting.md`。
2. **「本机无 gcc」**：当时 shell 进程启动早于 gcc 安装，继承了旧 PATH。gcc 其实已在系统 PATH 中。遇到 `-race requires cgo` 先重开终端。
